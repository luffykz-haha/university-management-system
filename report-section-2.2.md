# Section 2.2 – Test Automation

## Overview

Test automation for the University Management System was implemented using JUnit 5 as the primary unit testing framework, Mockito for dependency mocking, and JaCoCo (Java Code Coverage) for measuring and reporting code coverage. All of these tools are integrated directly into the Maven build lifecycle, meaning tests are discovered, executed, and reported automatically whenever the developer runs a build — no separate configuration or manual steps are required.

## Framework and Tools

The test automation stack is composed of four main tools. JUnit 5 (Jupiter), provided through the Spring Boot 3.4.4 Bill of Materials, serves as the foundation for writing and executing all unit tests. Mockito, also included transitively via the `spring-boot-starter-test` dependency, is used to create mock substitutes for external dependencies such as JPA repositories and the JWT utility, allowing each service to be tested in complete isolation. The Maven Surefire Plugin version 3.5.2 is responsible for discovering and running the test classes during the Maven `test` phase and writing the results as both XML and plain-text report files. Finally, the JaCoCo Maven Plugin version 0.8.12 instruments the compiled bytecode, collects execution data while the tests run, and then generates a detailed HTML, XML, and CSV coverage report automatically at the end of every `mvn test` invocation.

Because the application normally connects to a live PostgreSQL database, a dedicated test configuration file was added at `src/test/resources/application.properties`. This file substitutes the PostgreSQL driver with an H2 in-memory database for the duration of the test run, ensuring that all tests — including the Spring Boot context-load smoke test — can execute in any environment without requiring an external database server to be running.

## How Tests Are Structured

All unit tests reside under `src/test/java/org/example/ums/` and mirror the package structure of the main source code. Tests for the three service classes are grouped under the `service` sub-package, while tests for the JWT utility class are placed under the `util` sub-package. The existing `UniversityManagementSystemApplicationTests` class at the root of the test package serves as a smoke test, verifying that the full Spring Boot application context loads without errors.

Each service test class is annotated with `@ExtendWith(MockitoExtension.class)`, which activates the Mockito framework without starting the Spring container. This keeps individual tests extremely fast and fully isolated from one another. Dependencies such as repositories and the JWT utility are declared as `@Mock` fields, and Mockito automatically creates lightweight stub implementations of them. The service class under test is annotated with `@InjectMocks`, which causes Mockito to instantiate it and inject the mock dependencies via constructor injection. Assertions throughout all test classes use AssertJ's fluent API, employing `assertThat` for positive assertions and `assertThatThrownBy` to verify that the correct exceptions are thrown under invalid input or insufficient permissions.

## Test Cases Written

A total of 45 new unit tests were written across four test classes, covering the full range of success paths, validation failures, and authorization boundary conditions in the application.

The `UserServiceTest` class contains 12 tests targeting the `UserService`. These tests verify that retrieving all users returns the complete repository result, that fetching a user by a valid identifier returns the correct entity, and that fetching an unknown identifier throws a `NotFoundException`. The tests also confirm that creating a user automatically assigns the `ROLE_STUDENT` role and persists the entity. For the credential update operation, tests ensure that a user who provides the correct password can update their own name, that a wrong password raises a `ValidationException`, and that a non-admin user attempting to update another user's credentials receives a `ForbiddenAccessException`. The role update tests confirm that only an admin can change a user's role, while the delete tests verify that a user can remove their own account, that an admin can remove any account, and that a non-admin attempting to delete a different user is rejected.

The `CourseServiceTest` class contains 11 tests targeting the `CourseService`. These cover retrieval of all courses, fetching a course by a valid or invalid identifier, and creation of a course by both an admin and a teacher. A test also confirms that a student attempting to create a course is denied access with a `ForbiddenAccessException`. The update tests verify that both title and description are replaced when new values are provided, that blank fields fall back to the existing course values, and that a student cannot perform an update. The delete tests confirm that an admin can remove a course while a student cannot.

The `EnrollmentServiceTest` class contains 12 tests targeting the `EnrollmentService`. These verify that an admin retrieving all enrollments receives the complete list, while a student receives only the enrollments associated with their own account. Tests for retrieval by identifier confirm that an admin can access any enrollment, that a student can access their own enrollment, and that a student attempting to view an enrollment belonging to another student receives a `NotFoundException`. The creation tests verify that both admins and teachers can enroll a student in a course, that a non-existent student identifier raises a `NotFoundException`, and that a student role is forbidden from creating enrollments entirely. The delete tests confirm the same role-based restrictions.

The `JwtUtilTest` class contains 10 tests that exercise the `JwtUtil` class without any mocking, using the real signing key and JWT library. These tests confirm that token generation produces a non-empty, well-formed JWT string, that tokens generated for different users are distinct, that the email address embedded in the token payload can be correctly extracted, that the assigned roles are retrievable and accurate for both admin and student users, and that `extractSingleUserRole` returns the first role in the list. The validation tests confirm that a freshly generated token is valid for its owner, that the same token fails validation for a different user, and that passing a malformed string to the validator raises an exception.

## Test Execution Results

All tests are run by executing `mvn clean test` from the project root, or by double-clicking the provided `test-automation.bat` script. Maven Surefire picks up all test classes automatically and writes individual result files to `target/surefire-reports/`. The final execution produced the following results:

`UserServiceTest` ran 12 tests in 0.067 seconds with zero failures. `CourseServiceTest` ran 11 tests in 3.695 seconds with zero failures. `EnrollmentServiceTest` ran 12 tests in 0.334 seconds with zero failures. `JwtUtilTest` ran 10 tests in 0.403 seconds with zero failures. The Spring Boot context load test ran 1 test in 13.38 seconds with zero failures. Across all test classes, a total of 46 tests were executed, with zero failures, zero errors, and zero skipped tests. The overall build result was `BUILD SUCCESS`.

## Code Coverage Report

JaCoCo collected execution data throughout the test run and generated a full coverage report at `target/jacoco-report/index.html`, with additional machine-readable versions in CSV and XML format at the same location. Coverage was measured specifically for the service and utility layers, which contain the core business logic of the application. Controllers, security filters, and generated mapper implementations were not targeted by unit tests and are therefore expected to show lower coverage; they would require integration or end-to-end tests to cover effectively.

For `CourseService`, instruction coverage reached 100%, meaning every instruction in the class was exercised by at least one test, with branch coverage at 83%. `UserService` achieved 88% instruction coverage and 75% branch coverage. `EnrollmentService` reached 77% instruction coverage and 72% branch coverage. `JwtUtil` achieved 96% instruction coverage and 60% branch coverage. Across the entire service layer combined, the tests covered approximately 85% of instructions and 76% of branches, demonstrating thorough validation of the business rules implemented in the application.

## How to Run Test Automation

A Windows batch script named `test-automation.bat` is provided at the project root. When executed, it runs `mvn clean test`, which compiles the source and test code, executes all test classes via Maven Surefire, and triggers JaCoCo to generate the coverage report at the end of the test phase. The script then inspects the Maven output to determine whether the build succeeded or failed, prints a human-readable summary to the console, and displays the paths to all generated reports. Developers may also run the equivalent Maven command directly — `./mvnw clean test` — to achieve the same result. After the run completes, the HTML coverage report can be opened by navigating to `target/jacoco-report/index.html` in any web browser, and the raw Surefire test result files are available under `target/surefire-reports/`.
